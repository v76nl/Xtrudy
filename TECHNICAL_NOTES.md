# Technical Notes: Web 3D & 3D Printing Compatibility

> Distilled from the development of **xtrudy** — a browser-based 2D-to-3D keychain generator.
> Topics: Three.js, opentype.js, Clipper2, STL export, slicer behavior, manifold geometry.

---

## 1. Coordinate System Differences

### Three.js vs OpenType / SVG

| System | Y-axis direction |
|--------|-----------------|
| Three.js (and WebGL) | **Y-up**: Y increases upward |
| OpenType / SVG | **Y-down**: Y increases downward |

When you take glyph path commands from `opentype.js` (`font.getPath(...)`) and feed them
directly into `THREE.ExtrudeGeometry`, the resulting shape is **vertically mirrored** —
text appears upside-down.

**Fix:** Apply `geometry.scale(1, -1, 1)` after extrusion. However, a negative scale flips
the winding order of all triangles, which inverts face normals. You must reverse the triangle
index order immediately after to restore correct normals.

```javascript
// flipYCorrectly: scale Y, then re-wind triangles to fix normals
function flipYCorrectly(geometry, scaleX = 1, scaleY = -1, scaleZ = 1) {
    geometry.scale(scaleX, scaleY, scaleZ);
    if (scaleX * scaleY * scaleZ < 0) {
        // For indexed geometry: swap index[i] <-> index[i+2] for each triangle
        // For non-indexed geometry: swap vertex 0 <-> vertex 2 in each triangle triple
        geometry.computeVertexNormals();
    }
}
```

> **Key rule:** Any time the product of all three scale components is negative, triangle
> winding is reversed and normals must be recomputed.

## 2. Three.js ExtrudeGeometry Internals

### Extrusion direction

`THREE.ExtrudeGeometry(shape, { depth: d })` extrudes **in the +Z direction**:

- Back cap (shape face): at `z = 0`, normal pointing **-Z**
- Front cap: at `z = depth`, normal pointing **+Z**
- Side walls: connecting back and front caps

### Caps and holes

If the `THREE.Shape` has `.holes` (via `shape.holes.push(holePath)`), `ExtrudeGeometry`:

- Triangulates the cap using `THREE.ShapeUtils.triangulateShape` (ear-clipping)
- Generates inner side walls for each hole boundary

### Side wall tessellation

Side walls are generated by sampling the shape's curves at `curveSegments` intervals (default 12).
Each sampled point creates a quad (2 triangles) connecting `z=0` to `z=depth`. This means:

- **The XY coordinates of side wall vertices = the shape's sample points**
- If two shapes share the same boundary points (e.g., from the same Clipper output), their
  side walls will be at **exactly the same XY positions** — which can cause coincident geometry
  if both are extruded over the same Z range.

## 3. Font Path to 3D Pipeline

### opentype.js path commands

`font.getPath(char, x, y, size).commands` returns an array of:

| Command | Meaning |
|---------|---------|
| `M x y` | Move to (start new sub-path) |
| `L x y` | Line to |
| `Q x1 y1 x y` | Quadratic Bezier |
| `C x1 y1 x2 y2 x y` | Cubic Bezier |
| `Z` | Close path |

Each `M` command starts a new contour. A single glyph (e.g., "A") typically has:

- 1 outer contour (the letter outline)
- 1 or more inner contours (the holes/counters, e.g., the triangle inside "A")

### Converting to THREE.Shape

```javascript
commands.forEach(cmd => {
    switch(cmd.type) {
        case 'M': currentShape = new THREE.Shape(); currentShape.moveTo(cmd.x, cmd.y); break;
        case 'L': currentShape.lineTo(cmd.x, cmd.y); break;
        case 'Q': currentShape.quadraticCurveTo(cmd.x1, cmd.y1, cmd.x, cmd.y); break;
        case 'C': currentShape.bezierCurveTo(cmd.x1, cmd.y1, cmd.x2, cmd.y2, cmd.x, cmd.y); break;
        case 'Z': currentShape.closePath(); break;
    }
});
```

### Hole assignment

Raw shapes from a glyph need to be classified as **solid** or **hole** based on the
**signed area** of their point polygon:

```javascript
const area = THREE.ShapeUtils.area(pts);
// Positive area => counter-clockwise => solid (outer contour)
// Negative area => clockwise => hole (inner contour)
```

After Clipper2 Boolean Union, the largest-area contour is the primary solid; smaller contours
of opposite sign are holes. Each hole should be assigned to its enclosing solid (use bounding
box centroid containment check).

## 4. Clipper2 (clipper2-js) for Boolean Path Operations

### Why it is needed

Fonts like "X", "O", "B" have **self-intersecting** or **nested** sub-paths. If you extrude
them naively, overlapping regions produce internal walls inside the solid => non-manifold
geometry => slicer errors.

### Integer coordinate system

Clipper2 works exclusively with **64-bit integers**. Float coordinates must be scaled:

```javascript
const CLIPPER_SCALE = 1e6; // 1 unit = 1 micrometer

// Float -> Clipper integer
{ x: Math.round(floatX * CLIPPER_SCALE), y: Math.round(floatY * CLIPPER_SCALE) }

// Clipper integer -> Float
floatX = pt.x / CLIPPER_SCALE;
```

### Key operations

```javascript
import { Clipper, Paths64, FillRule } from 'clipper2-js';

// Union: merge all sub-paths into clean non-self-intersecting outlines
const unified = Clipper.Union(paths, undefined, FillRule.NonZero);

// Difference: subtract shape B from shape A
const result = Clipper.Difference(pathsA, pathsB, FillRule.NonZero);
```

### FillRule.NonZero

- **NonZero**: a point is inside if the winding number is not 0 (standard for fonts)
- **EvenOdd**: alternates inside/outside with each crossing (not suitable for most fonts)

### Typical font pipeline with Clipper

1. Parse all character paths => raw sub-paths (no hole assignment yet)
2. Convert all sub-paths to `Paths64`
3. `Clipper.Union(allPaths, undefined, FillRule.NonZero)` => clean outlines
4. `paths64ToThreeShapes()` => classify solids vs holes by area sign
5. Feed cleaned `THREE.Shape[]` to `ExtrudeGeometry`

## 5. Manifold Geometry: Definition and Requirements

### What is a manifold mesh?

A **manifold** (or "watertight") mesh satisfies:

1. Every edge is shared by **exactly 2** triangles
2. All triangles around a vertex form a single connected "fan" (no branching)
3. No face normals point inward (consistent outward winding)

### What makes a mesh non-manifold?

| Problem | Description | Slicer effect |
|---------|-------------|---------------|
| **T-junction** | An edge from mesh A touches the interior of a face of mesh B | Hole in print |
| **Coincident opposite-face** | Two triangles at the same position with opposite normals | Void region |
| **Non-manifold edge** | 3+ triangles share the same edge | Inside/outside undefined |
| **Open boundary** | An edge has only 1 adjacent triangle | Hole in shell |

### How slicers handle non-manifold geometry

Slicers use **ray casting** or **winding number** algorithms to determine inside vs outside
at each layer. Coincident opposite-facing triangles cause the ray-crossing count to cancel
=> the slicer treats that region as **outside** => prints nothing (void).

## 6. The Shared-Face / Coincident-Wall Problem

### The specific bug encountered in xtrudy

**Buggy design:**

- Text column: XY = text footprint, Z = `-baseThickness` to `+modelThickness`
- Base ring (donut): XY = `base outline minus text footprint`, Z = `-baseThickness` to `0`

Both geometries generate side walls at the text boundary from `z = -baseThickness` to `z = 0`:

| Geometry | Side wall position | Normal direction |
|----------|--------------------|-----------------|
| Text column outer wall | Text footprint boundary | Points outward |
| Base ring inner wall (hole wall) | Same text footprint boundary | Points inward |

=> Two sets of triangles at **identical XYZ positions** with **opposite normals**.

### Why mergeGeometries + mergeVertices made it worse

```javascript
const merged = mergeGeometries(geometries, false); // concatenate all triangles
const cleaned = mergeVertices(merged, 1e-4);        // merge duplicate vertices
```

After `mergeVertices`, the opposing triangles now **share vertices** => non-manifold edge
(3+ triangles share one edge). The slicer sees this as a void in the printed model.

### The fix: separate Z ranges completely

```
After fix:
z = +modelThickness  +----------+
                     | Text     |  <- Shell 1 (z=0 to +model)
z = 0           -----+          +--------- (edge-only contact, no face overlap)
                +----+----------+----+
                |  Base ring (donut) |  <- Shell 2 (z=-base to 0)
z = -baseThick  +--------------------+
```

- Text column: `depth = modelThickness` only, starts at z=0, no Z translate needed
- Base ring: unchanged (z=-base to 0)
- No Z range overlap => **no coincident side walls**

### Export: do not merge shells

```javascript
// WRONG: merging creates coincident walls => non-manifold
const merged = mergeGeometries(geometries);
const cleaned = mergeVertices(merged);

// CORRECT: export rootGroup directly, let the slicer union the shells
const result = exporter.parse(rootGroup, { binary: true });
```

## 7. Multiple Shells in STL Files

### STL format basics

Binary STL is a flat list of triangles. There is no concept of "shells" or "objects" —
it is just a bag of triangles. Multiple meshes from a Three.js scene all get exported
into one list.

```
[80-byte header][uint32: triangle count]
[per-triangle: normal(3xfloat32) + v1(3xfloat32) + v2(3xfloat32) + v3(3xfloat32) + attrib(uint16)]
```

### Slicer interpretation of multiple shells

Modern slicers (ideamaker, PrusaSlicer, Cura) interpret multiple shells via **Boolean Union**:

- Each closed surface is detected as a separate shell
- The union of all shells is taken as the printable solid
- Adjacent shells sharing only edges (not faces) are correctly unioned

This is why exporting without `mergeGeometries` works: each part remains a clean
independent shell, and the slicer correctly takes their union.

### When merging IS useful

Merging is beneficial **only** when:

- The meshes do NOT share any boundary geometry (completely separate, non-touching)
- You want to reduce the shell count for slicer performance
- You are using `mergeVertices` to resolve T-junctions where vertex positions are known
  to be exactly coincident

## 8. STL Export from Three.js (STLExporter)

### exporter.parse(object, { binary: true })

- Traverses the scene graph and includes all `Mesh` objects
- For each triangle, applies the mesh's `matrixWorld` to transform to world space
- Computes face normals from vertex positions using cross product: `normal = (b-a) x (c-a)`

### Negative scale (mirrorX) gotcha

When a `Group` has `scale.x = -1` (e.g., for stamp/hanko mirror):

- `applyMatrix4(matrixWorld)` bakes the negative scale into vertex positions
- This **reverses triangle winding**
- The STLExporter recomputes normals from the reversed winding => normals point inward
- Most slicers have an "auto-fix normals" option, but it is not guaranteed

**Mitigation:** After applying a negative-scale matrix to a cloned geometry, reverse triangle
indices before export. Or rely on the slicer's normal-fixing feature.

### computeVertexNormals() before export

Calling `geometry.computeVertexNormals()` before STL export has **no effect on the STL normals**.
The `STLExporter` recomputes per-face normals from vertex positions regardless. It only affects
Three.js viewport rendering (smooth shading).

## 9. Ring / Torus Geometry for 3D Printing

### Hollow cylinder (printable strap ring)

Instead of `THREE.TorusGeometry` (solid torus with circular cross-section), a hollow cylinder
is created using an annular 2D cross-section extruded in Z:

```javascript
const shape = new THREE.Shape();
shape.absarc(0, 0, outerR, 0, Math.PI * 2, false); // outer circle (CCW)
const hole = new THREE.Path();
hole.absarc(0, 0, innerR, 0, Math.PI * 2, true);    // inner circle (CW = hole)
shape.holes.push(hole);
const geometry = new THREE.ExtrudeGeometry(shape, {
    depth: cylHeight, bevelEnabled: false, curveSegments: 32
});
```

This produces a manifold hollow cylinder with flat top/bottom faces — more suitable for
printing than `TorusGeometry` whose cross-section is circular.

### Ring reinforcement plate

A connecting "fin" between the ring and the base plate. Shape is computed in the ring's
local 2D space:

- **Case A:** Base top edge intersects the ring's outer circle => arc + chord shape
- **Case B:** Base top edge is entirely below the ring => rectangular + half-arc shape

## 10. Z-Positioning Strategy

### Coordinate conventions

| Part | Z range | Notes |
|------|---------|-------|
| Text column | `0` to `+modelThickness` | Sticks up above base |
| Base ring (donut) | `-baseThickness` to `0` | Below the top surface |
| Ring (hollow cylinder) | Centered at `-baseThickness / 2` | Middle of base Z range |
| Ring reinforcement | Same center Z as ring | Must match ring |

Placing the base at negative Z and the text at positive Z means `z=0` is the top surface
of the base / bottom surface of the text. This is a natural reference plane and avoids
any Z-overlap between the two parts.

## 11. THREE.ShapeUtils.area() for Winding Detection

```javascript
const area = THREE.ShapeUtils.area(points); // points: THREE.Vector2[]
// Positive area => counter-clockwise => solid (outer contour in Three.js convention)
// Negative area => clockwise => hole (inner contour)
```

After Clipper2 Union, the outermost contour's winding is determined by Clipper's output
convention. Always measure the sign of the first (largest-area) contour empirically and
treat it as the "solid" sign.

## 12. Summary: Rules for Manifold 3D-Printable Geometry

1. **No face sharing between adjacent solids.** Two solids that touch should share only
   edges, never faces at the same Z plane with the same XY region.

2. **No coincident opposite-facing triangles.** If two triangles occupy the same plane with
   opposite normals, slicers treat that region as void.

3. **Separate Z ranges for separate parts where possible.** The cleanest way to avoid
   coincident walls is to not overlap the Z ranges of adjacent extrusions.

4. **Export as multiple independent shells rather than merging.** `mergeGeometries +
   mergeVertices` is only safe when the input meshes have no shared boundary geometry.

5. **Use Clipper2 Union before extrusion for font paths.** Font sub-paths may
   self-intersect. Boolean union in 2D is cheaper and more reliable than fixing the
   3D mesh afterward.

6. **Correct winding after any negative scale.** Flip triangle index order whenever
   the product of all three scale factors is negative.

7. **Test STL in slicer, not just Three.js viewport.** Three.js uses DoubleSide material
   which hides winding/normal errors. The slicer is strict about normals.
